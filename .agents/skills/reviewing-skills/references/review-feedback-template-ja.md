## チェックリスト詳細（チェックあり・なし）

記入ルール:
- `[x]` = 適合（チェックあり）
- `[ ]` = 未適合または未確認（チェックなし）

### 1. Metadata Quality

- [ ] `name` は小文字・数字・ハイフンのみ
- [ ] `name` は64文字以下
- [ ] `name` に `anthropic` / `claude` を含まない
- [ ] `description` は具体的かつ空でない
- [ ] `description` に「何をするか」と「いつ使うか」が含まれる
- [ ] `description` は三人称で書かれている

### 2. SKILL.md Quality

- [ ] 指示が簡潔で冗長な背景説明がない
- [ ] 本文が500行以内、または詳細を参照ファイルへ分割済み
- [ ] 脆弱なタスクで主ワークフローが明確かつ順序立っている
- [ ] タスク特性に応じた自由度設定になっている

### 3. Progressive Disclosure

- [ ] `SKILL.md` は概要中心で必要時のみ詳細参照に誘導する
- [ ] 参照ファイルは `SKILL.md` から直接リンクされている（1レベル）
- [ ] 100行超の参照ファイルに目次がある
- [ ] ディレクトリ/ファイル名が説明的

### 4. Workflow and Feedback Loops

- [ ] 複数ステップ作業にチェックリストまたは順序付き手順がある
- [ ] 破壊的・高リスク操作の前に検証ステップがある
- [ ] 品質確保の反復ループ（検証→修正→再検証）がある

### 5. Script and Tooling Quality

- [ ] スクリプトが想定エラーを明示的に処理している
- [ ] 定数やデフォルト値に根拠がある（マジックナンバー回避）
- [ ] 必要依存関係が明示されている
- [ ] スクリプトを「実行する/参照として読む」が明確
- [ ] パス表記がフォワードスラッシュで統一されている

### 6. Content Risks and Anti-Patterns

- [ ] 時間依存情報が避けられている（必要ならレガシー節へ分離）
- [ ] 用語が一貫している
- [ ] 不要に選択肢を増やしすぎていない
- [ ] 出力品質が重要な箇所で具体例が示されている

### 7. Testing Evidence

- [ ] 代表的評価シナリオが最低3件ある
- [ ] 想定モデル層で挙動検証されている
- [ ] 実運用フィードバックまたは改善履歴がある
